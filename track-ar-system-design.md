# TrackAR — 系统技术设计文档

## 1. 核心目标

8 条跑道、8 名运动员的 **100m 直线** 或 **400m 椭圆** 赛道直播画面中，在**不遮挡任何运动员**的前提下，为每一名运动员实时叠加：
- 当前名次（Rank）
- 当前成绩/时间（Time）
- 图形随运动员一起沿跑道前进

支持相机在比赛期间的任意运动（摇镜/俯仰/变焦/推拉/升降），AR 叠加自动跟随。

---

## 2. 零遮挡的核心原则

```
┌──────────────────────────────────────────────────────┐
│  每条跑道 = 约束走廊                                   │
│  跑道内部分为三个区域：                                  │
│                                                        │
│  [前方空域] ← 运动员 ← [后方空域]                     │
│                                                        │
│  → 图形固定放在 "前方空域"（运动员到终点之间）          │
│  → 或放在 "后方空域"（起点到运动员之间）               │
│  → 绝不覆盖运动员所在区域                               │
└──────────────────────────────────────────────────────┘
```

### 禁止性规则（硬约束）

| # | 规则 | 违反后果 |
|---|------|----------|
| 1 | 同一跑道内，图形 bounding box 不得与运动员 bounding box 在图像平面上有交集 | 直接遮挡 |
| 2 | 图形不得跨出本跑道边界 | 侵入邻道，可能遮挡邻道运动员 |
| 3 | 远侧跑道（透视下较小）的图形不应在图像空间上叠加到近侧跑道的运动员身上 | 透视遮挡 |

## 3. 系统架构总览

```
视频输入 (File / Camera)
         ↓
┌──────────────────────────────────────┐
│          预处理 / 缩放                │
└──────────────────────────────────────┘
         ↓
┌──────────────────┐  ┌──────────────────────────────┐
│ 相机校准 (PnP)    │  │ 目标检测 (YOLOv8)            │
│ + KLT 实时追踪    │  │ + LaneAssigner + 卡尔曼      │
└──────────────────┘  └──────────────────────────────┘
         ↓                       ↓
┌──────────────────────────────────────────┐
│ 位置估计 → (lane#, d_m_from_start)       │
└──────────────────────────────────────────┘
         ↓
┌──────────────────────────────────────────┐
│ 排名 & 计时 (RaceTimer)                   │
└──────────────────────────────────────────┘
         ↓
┌──────────────────────────────────────────┐
│ Occlusion Guard — 安全锚点计算            │
└──────────────────────────────────────────┘
         ↓
┌──────────────────────────────────────────┐
│ 图形渲染 + Standings 面板 (OpenCV)       │
└──────────────────────────────────────────┘
         ↓
       输出帧
```

### 相机追踪与运动员追踪独立

KLT 特征追踪器仅操作**背景特征**（运动员特征被 USAC_MAGSAC 作为外点剔除）。单应性矩阵送入 `Projector.track_homography()`，将标定参考点通过当前外参投影，经单应性矩阵变换后重新求解 PnP，产生**无漂移的 6-DOF 位姿更新**。

## 4. 相机与视角设计

### 推荐机位

```
         ┌─────┐
         │ CAM │  ← 侧上方 15-25m 高，俯角 25°-40°
         └──┬──┘
            │
    ──── 跑道 1 ────  (近端)
    ──── 跑道 2 ────
    ──── 跑道 3 ────
    ──── 跑道 4 ────
    ──── 跑道 5 ────
    ──── 跑道 6 ────
    ──── 跑道 7 ────
    ──── 跑道 8 ────  (远端)
    ================= 终点线

            ↑
         起跑线
```

**视角选择理由**：
- 侧上方俯角 → 每条跑道的视觉分离度最大，透视重叠最小
- 每条跑道在图像中形成近似平行的梯形区域
- 运动员沿跑道方向运动，图形沿同一方向跟随

### 备用机位

| 视角 | 优点 | 遮挡风险 | 推荐度 |
|------|------|----------|--------|
| 侧上方（推荐） | 跑道分离好，遮挡最小 | 低 | 高 |
| 正侧面（平视） | 运动员轮廓清晰 | 中—前后跑道的运动员可能在图像中重叠 | 中 |
| 正上方（天眼） | 绝对无遮挡 | 无—但看不到运动员侧面，观赏性差 | 中 |
| 终点正对 | 冲刺画面震撼 | 极高—8 人前后重叠 | 低 |

### 动态相机模式

支持 follow-mode：相机自动跟随运动员平均位置，在领先运动员距终点 < 15m 时锁定终点。

## 5. 相机校准

### 5.1 已知几何先验

#### 100m 直线

| 参数 | 值 |
|------|-----|
| 跑道长度 | 100.00 m |
| 跑道宽度 | 1.22 m |
| 跑道总数 | 8 |
| 跑道总宽 | 9.76 m |

#### 400m 椭圆 (IAAF 标准)

| 参数 | 值 |
|------|-----|
| 内道内径 | 36.50 m |
| 跑步线偏移 | 0.30 m |
| 直线段长度 | 84.39 m |
| 跑道宽度 | 1.22 m |
| 每道弧长 | `π × (36.5 + 0.3 + (lane-1) × 1.22)` |
| 总弧长 | `2 × 84.39 + 2 × 弧长` |
| 梯形起跑偏移 | `总弧长 - 400`（每道不同） |
| 每道终点距离 | `400 - 梯形起跑偏移`（不同车道终点不同） |

### 5.2 坐标系定义

```
世界坐标系 (WCS):
  X: 沿跑道方向（起点→终点）
  Y: 垂直跑道方向（跑道 1→跑道 8）
  Z: 垂直地面向上
  原点: 跑道 1 与起跑线交点（100m）/ 弯道圆心（400m）

像平面坐标系 (ICS):
  (u, v): 像素坐标

相机坐标系 (CCS):
  (Xc, Yc, Zc): 相机为中心
```

### 5.3 校准方式

使用 **PnP (Perspective-n-Point)** 求解相机位姿。

#### 标准模式（4 点标定）

操作员在画面上点选 4 个参考点：
- 起点线与跑道 1、跑道 8 的交点（2 个点）
- 终点线与跑道 1、跑道 8 的交点（2 个点）

（100m 和 400m 共用此方式；400m 的"起点"和"终点"指各自跑道的起跑线和终点线世界坐标。）

#### 标定物模式（远摄镜头推荐）

当起终点不在同一画面时，在跑道上放置已知尺寸的标定物（A4 纸、纸箱等），设定其位置（距离标记 dm、车道号）和尺寸，点击 4 个角点完成标定。

**400m 曲线处理**：`TrackGeometry.calibration_target_points()` 在曲线段使用切线空间计算标定物 4 角，通过 `world_coord(lane, dm±ε)` 求沿跑道与垂直方向单位向量，构建与跑道严格对齐的矩形。解决了 naive dm/y 坐标在曲线上与真实世界坐标偏差达 282m 的 PnP 退化问题。

#### 跨帧标定

当 4 个点分布在不同帧（如起跑线在一帧、终点线在另一帧），使用 **ORB 特征匹配 + FrameTracker** 将后点击的点配准到首帧坐标系，统一求解 PnP。

> **标定只需一次**。之后相机可以任意运动（摇镜/俯仰/变焦/推拉/升降），KLT 追踪器持续跟踪跑道表面特征，通过 `track_homography()` 逐帧更新 6-DOF 位姿。

## 6. 运动员检测与跟踪

### 6.1 检测模型

| 组件 | 方案 |
|------|------|
| 模型 | YOLOv8s (Small) — 默认；支持 n/m 型号 |
| 类别 | person (COCO 预训练) |
| 加速 | PyTorch CUDA (cuDNN) |
| 输入 | 1080p → 缩放到 640x640 |
| 吞吐 | ~115 fps (RTX 5070) |
| 降级 | DummyDetector（合成场景 / 无 GPU 环境） |

### 6.2 身份分配策略（核心）

**核心思想：不用视觉 ReID，用几何约束做硬绑定。**

```
每帧得到 N 个 bbox:
  1. 对每个 bbox，取其底部中点 (u, v_bottom) 作为脚底位置
  2. 通过单应性逆投影到跑道平面 → 得到世界坐标 (X, Y)
  3. 根据 Y 值判定属于哪个跑道: lane = floor(Y / lane_width)
  4. NMS (IoU ≥ 0.85) 抑制同运动员重复检测
  5. 2 帧待定确认制: 连续 2 帧匹配同一跑道才建立跟踪
  6. 卡尔曼预测引导搜索: 在当前帧预测位置附近匹配
  7. 约束: 不允许 ID 跨跑道跳变
```

**100m 额外过滤**：标定时计算跑道在图像空间的包围盒，检测框中心落在此包围盒外则视为观众/工作人员，直接丢弃。

**400m 回收购**：丢失跟踪后，使用宽松阈值在全局范围重新搜索收购。

**鲁棒性保证**：
- 卡尔曼滤波对每个跑道单独运行（3 状态：位置/速度/加速度）
- 检测丢失时，卡尔曼 coast 最多 600 帧（10 秒 @ 60fps）
- coast 期间位置限制 ≥ 0，速度限制 ±15 m/s
- `coast_count` 累积未匹配帧数，逐帧递增

### 6.3 位置估计

得到运动员在世界坐标系中的沿跑道距离：

```
d_m = 沿跑道距离（起跑线到运动员投影点的弧长）
```

100m：直线距离 = X 坐标
400m：弧长距离，按车道不同经过弯道展开

同时记录：
- `frame_count`: 帧计数
- `timestamp`: 视频时间戳（用于计时）
- `speed_mps`: 瞬时速度（卡尔曼估计）
- `confidence`: 跟踪置信度

## 7. 排名与计时

### 7.1 实时排名

```
rank = sorted(positions, key=lambda p: p.d_m, reverse=True)
```

按 `d_m` 从大到小排序（领先者 d_m 最大，因为是从起跑线沿跑道的距离）。
起跑瞬间所有运动员 `d_m ≈ 0`，排名按检测先后微调。

### 7.2 计时

使用**视频时间戳**（非系统时钟），确保与视频帧率同步：

- 起跑检测：**≥ 2 名运动员 `d_m > 0.5m`** 时触发统一发令计时
- 完赛记录：运动员 `d_m >= finish_distance(lane) - 0.5` 时记录终帧时间
- 100m：所有车道 `finish_distance = 100m`
- 400m：每道不同 `finish_distance = 400 - stagger_offset(lane)`
- 整体计时器在所有 8 道完成后停止

## 8. Occlusion Guard — 核心安全机制

这是本系统的核心机制，保证**任何条件下图形不遮挡运动员**。

### 8.1 运动员占据区域建模

对于运动员，定义其在图像平面上的**占据区域**：

```
O_i = {
    bbox: [u_left, v_top, u_right, v_bottom],   // 检测框
    foot_pos: (u_foot, v_foot),                  // 脚底位置（投影到地面）
}
```

### 8.2 图形安全锚点计算

图形放置在运动员**前方**（跑向终点方向），设定 3D 偏移量：

```
世界坐标系中:
  图形锚点 G_i_world = (X_i + offset_ahead, Y_lane_center, 0)

  其中:
    X_i = 运动员 i 的当前沿跑道距离
    offset_ahead = 2.0 米 (可配置)
    Y_lane_center = 跑道 i 的中心线 Y 坐标

投影到图像:
  G_i_image = project(G_i_world)
```

### 8.3 安全碰撞检测（每帧执行）

```
for each lane i:
  1. 计算 G_i_image (图形锚点投影)
  2. 计算 O_i_image (运动员占据区域)
  3. 判断图形 bbox 是否与 O_i_image 重叠:
     if collision:
       尝试以下策略 (按优先级)：
         a. 增大 offset_ahead (加大前方距离)
         b. 切换到 "后方" 放置: offset = -1.0m
         c. 微调水平偏移 (靠跑道内侧/外侧)
         d. 微调垂直偏移 (抬高)
      每次调整后重新检查碰撞，直到无碰撞
  4. 跨跑道交叉检查:
     for each (lane_i, lane_j):
       检查 G_i_image 是否与 O_j_image 在图像上重叠
       if cross_collision:
         微调 G_i 的水平位置
```

### 8.4 深度层级（Z-ordering）

渲染时按照距相机的深度排序。世界空间深度由 `DepthSorter` 计算：

```
渲染顺序 (从远到近):
  1. 跑道地面
  2. 远端跑道图形
  3. 远端跑道运动员
  4. 近端跑道图形
  5. 近端跑道运动员
```

### 8.5 安全边界冗余

- 图形与运动员之间保留 **至少 0.5m（3D 世界空间）** 的物理间隔
- 图形高度限制在跑道平面的 **±0.3m（世界空间）** 以内
- 图形水平宽度不超过 **0.8 × 跑道宽度（~0.98m）**

## 9. 图形渲染设计

### 9.1 图形内容

```
┌──────────────────┐
│ ①  #3            │  ← 名次 (大号数字)
│  08.45s          │  ← 时间成绩
│  [RUNNER NAME]   │  ← 运动员名称（可选）
└──────────────────┘
```

### 9.2 渲染管线

纯 OpenCV 渲染（无 OpenGL/Three.js）：

1. `GraphicContent` 生成 RGBA 纹理（含名次/时间/姓名 + 跑道色边框）
2. `OcclusionGuard` 计算安全图像锚点
3. `DecalRenderer` 将纹理通过透视投影缩放到图像空间，Alpha 混合叠加到画面
4. `StandingsPanel` 绘制左上角排行榜面板（计时器 + 名次表）
5. `DebugOverlay` 可选绘制检测框和锚点

### 9.3 地面贴花方案

图形渲染为跑道地面上的 textured quad：
- 四边形贴合跑道平面
- 跟随运动员前进（offset_ahead）
- 透视畸变自然由投影处理
- `compute_graphic_bbox()` 计算图形四角在世界空间的 3D 坐标后投影到图像

**优点**：深度测试自动处理；运动员跨过图形时自然覆盖；视觉融合度好。

## 10. 极端情况处理

| 场景 | 风险 | 应对 |
|------|------|------|
| **运动员摔倒** | 其他运动员赶上，图形覆盖倒地者 | `EdgeCaseDetector` 检测 bbox 高度异常（< 50% 正常）或速度骤降 → 报警 |
| **终点冲刺（并排）** | offset_ahead 空间不足 | 自动切换为 offset_behind，图形放在运动员身后 |
| **运动员大幅领先** | 前方空域巨大 | offset_ahead 动态自适应 |
| **起跑瞬间（d_m ≈ 0）** | dm_movement 过滤导致运动员被丢弃 | `dm_movement < 1` 拒绝逻辑已移除 |
| **检测丢失** | 跟踪中断 | 卡尔曼 coast 最多 600 帧，位置/速度钳制 |
| **观众/工作人员误检** | 产生虚假跟踪 | 100m 模式使用 track_bbox 过滤；NMS 消除同人重复检测 |
| **相机抖动** | 图形位置跳动 | 位置 3 帧加权滑动平均（`PositionSmoother`） |

### 13.2 自动化测试套件

| 类别 | 数量 | 说明 |
